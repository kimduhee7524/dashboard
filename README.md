# 📊 광고 캠페인 대시보드

이 프로젝트는 플랫폼별 광고 캠페인 성과 데이터를 시각화하고 관리할 수 있는 대시보드 애플리케이션입니다.

---

## 🚀 1. 실행 방법

로컬 환경에서 프로젝트를 설치하고 실행하는 방법은 다음과 같습니다.

### 사전 요구 사항

- **Node.js** 22.13.1
- **pnpm** 10.8.1

### 설치 및 실행

```bash
# 1. 의존성 설치 (pnpm 권장)
pnpm install

# 2. 로컬 개발 서버 실행
pnpm dev
```

> **참고**: 이 프로젝트는 백엔드 서버 없이 동작하도록 구성되어 있으며, 로컬 개발 환경에서는 **MSW(Mock Service Worker)**를 활용하여 가상의 API 통신 환경을 제공합니다.

## 🛠 2. 기술 스택 및 선택 근거

프로젝트 요구사항 달성과 확장성을 고려하여 다음 기술 스택을 채택했습니다.

| 기술                             | 선택 근거 (Trade-off 및 이점)                                                                                                                                                                                                                                                                                                                                                                                                                         |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **React 19 + Vite + TypeScript** | React의 선언적 UI와 최신 생태계를 활용합니다. 빌드 속도가 빠른 Vite와 런타임 에러를 사전에 방지하는 TypeScript의 조합으로 DX(Developer Experience)와 안정성을 극대화했습니다.                                                                                                                                                                                                                                                                         |
| **Zustand** (전역 상태)          | 대시보드의 **글로벌 필터(날짜, 매체, 상태)**는 여러 위젯에서 공통으로 의존하는 상태입니다. Redux 대비 보일러플레이트가 훨씬 적고 직관적인 Zustand를 사용하여 가볍고 빠르게 전역 필터 스택을 관리합니다.                                                                                                                                                                                                                                               |
| **TanStack Query** (서버 상태)   | API 로딩, 에러, 캐싱 처리를 선언적으로 할 수 있도록 도입했습니다. 클라이언트 데이터(필터링, 정렬 등)와 서버 데이터를 명확히 분리하여 데이터 무결성을 보장합니다.                                                                                                                                                                                                                                                                                      |
| **TanStack Table**               | 데이터의 필터링, 정렬, 페이지네이션과 같은 복잡한 테이블 로직을 Headless 방식으로 제공합니다. UI와 로직이 결합되지 않아 대시보드의 요구사항에 맞게 테이블 스타일을 완전히 커스텀하기 매우 유리합니다.                                                                                                                                                                                                                                                 |
| **Recharts**                     | 차트 라이브러리 도입 전 대시보드의 데이터 규모를 최우선으로 고려했습니다. 초기 기본값이 1개월이고, 실무적으로도 캠페인 분석은 보통 1~3개월 또는 길어야 1년 단위로 이루어진다고 판단하였습니다. 본 과제의 데이터 역시 약 10개월 치 분량이므로, 이 정도 규모는 Canvas나 WebGL이 아닌 SVG 기반 차트로 렌더링해도 성능상 전혀 무리가 없다고 판단했습니다. 이에 React 환경에서 선언적으로 가장 빠르고 안정적인 구현이 가능한 Recharts를 최종 선택했습니다. |
| **React Hook Form + Zod**        | '캠페인 생성 폼' 등에서 복잡한 유효성 검사(Validation)를 최소한의 리렌더링으로 처리하기 위해 사용합니다. Zod를 통해 컴파일 타임의 타입 검사와 런타임 에러 검사를 폼 레벨에 결합했습니다.                                                                                                                                                                                                                                                              |
| **Tailwind CSS + Shadcn UI**     | 유틸리티 클래스를 통해 빠른 스타일링과 일관된 디자인 시스템(컬러, 타이포그래피)을 구축했습니다. Shadcn UI(Radix UI 기반)는 접근성(a11y)이 보장된 Headless 컴포넌트를 제공하여 품질 높은 UI를 손쉽게 개발하게 합니다.                                                                                                                                                                                                                                  |

## 🏗 3. 폴더 구조 및 아키텍처 (Layered Architecture)

프로젝트는 명확한 **관심사 분리(Separation of Concerns)**와 단방향 데이터 흐름을 위해 **Layered Architecture** 패턴을 차용하여 디렉토리를 분리했습니다.

```text
src/
├── api/             # 서버 API 호출 함수 계층 (Axios, Fetch 등)
├── domain/          # 비즈니스 로직 및 핵심 타입 (metrics 계산, 데이터 가공, 집계 등)
├── hooks/
│   ├── queries/     # 서버 데이터 접근 훅 (React Query)
│   └── derived/     # API 데이터 + 전역 필터 -> UI용 파생 데이터 생성 훅
├── store/           # 클라이언트 전역 상태 (Zustand: 필터 등)
├── components/      # View 계층 (UI 컴포넌트)
│   ├── ui/          # 재사용 가능한 순수 디자인 UI 컴포넌트 (Shadcn, Base UI)
│   ├── common/      # 도메인에 종속되지 않은 공통 모듈 (modal, form 등)
│   └── [feature]/   # 특정 도메인/기능별 컴포넌트 (campaign-table, platform-donut 등)
├── lib/             # 애플리케이션 공통 유틸리티 (포맷팅, 수학 함수 등)
└── pages/           # 라우팅 단위 페이지 컴포넌트
```

### 아키텍처 설계 의도

1. **로직과 UI의 분리**: `domain/` 디렉토리에 핵심 집계(aggregation)와 지표(metrics) 계산 로직을 두어, UI 뷰 컴포넌트가 무거워지는 것을 방지하고 단위 테스트를 용이하게 합니다.
2. **단방향 데이터 파이프라인**: `API (Raw Data)` -> `Hooks (Queries + Filters)` -> `Derived Data (데이터 병합 및 정렬)` -> `Components (UI)` 순서로 명확하게 데이터가 흐르도록 통제했습니다.

## 🧩 4. 컴포넌트 설계

재사용성과 확장성, 그리고 유지보수성을 극대화하기 위해 다음과 같은 컴포넌트 설계 패턴을 적용했습니다.

### 4.1. 합성 컴포넌트 (Compound Components) 패턴

모달(`FormDialog`)과 데이터 테이블(`DataTable`) 같이 내부 구조가 복잡하고 도메인에 따라 형태가 자주 변하는 UI 컴포넌트에 합성 컴포넌트 패턴을 적용했습니다.

- **모달 (`FormDialog`)**: `<FormDialog.Header>`, `<FormDialog.Body>`, `<FormDialog.Footer>` 등의 하위 컴포넌트로 분리하고 내부적으로 `react-hook-form`의 `FormProvider`를 공유하여, 사용하는 쪽에서 폼 렌더링을 매우 유연하게 조합할 수 있습니다.
- **데이터 테이블 (`DataTable`)**: Context API를 활용하여 최상단에서 한 번만 `table` 인스턴스를 주입받습니다. 하위의 `<DataTable.Toolbar>`, `<DataTable.Head>`, `<DataTable.Body>`, `<DataTable.Pagination>` 등의 요소들은 프롭스 드릴링(Prop Drilling) 없이 독립적으로 선언되어, 어떠한 레이아웃이나 요구사항 변화에도 쉽게 재사용 가능합니다.

### 4.2. UI 렌더링과 비즈니스 로직의 분리

대시보드의 복잡성을 제어하기 위해, **데이터 가공이 많이 필요한 컴포넌트들(`TrendChart`, `Top3Ranking`, `PlatformDonut` 등)을 기준**으로 UI 렌더링과 비즈니스 로직의 분리하였습니다. (단순한 공통 UI 컴포넌트는 오버엔지니어링을 피하기 위해 분리하지 않았습니다.)

### 4.3. 렌더링 최적화 및 사용자 경험(UX) 개선

- **불변성과 O(1) 탐색**: 필터링이나 데이터 집계 과정에서 불변성을 유지하고, 내부적으로 `Map`이나 `Set`을 적극 사용하여 대량의 데이터 매칭 시 시간 복잡도를 `O(1)` 수준으로 최적화했습니다.
- **선언적 뷰 (Declarative View)**: 복잡한 HTML 덩어리를 뷰 컴포넌트(`Top3RankingItem`, `PlatformPieChart` 등)로 잘게 분리하여 조건부 렌더링의 가독성을 크게 높였습니다.
